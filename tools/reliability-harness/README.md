# Reliability harness

Exposing functions to an agent is easy. The honest question is: *when a person asks for something
in their own words, how often does the agent pick the right function with the right arguments?* This
harness measures exactly that, and — because the whole premise of the project is that **a function's
KDoc is the prompt the agent reads** — it also measures how much the quality of those descriptions
moves the number.

## What it measures

For each natural-language intent the harness sends Gemini the five functions as tool declarations, a
fixed snapshot of the current task list (so id-based parameters are resolvable), and the user's
request. It records the function Gemini chose and the arguments it extracted, then scores them:

- **Function accuracy** — did it call the function the intent actually meant?
- **Restraint** — for out-of-scope requests ("what's the weather"), did it correctly call *nothing*
  instead of forcing a wrong call?
- **Parameter accuracy** — on correctly-routed write calls, did it extract the right title, task id,
  dependency, priority, due date, recurrence?

Each metric is reported overall and per goal.

## The KDoc experiment

The same dataset is run twice:

- **rich** — the real KDocs from the `agent/` functions, written as prompts.
- **terse** — the descriptions stripped to bare one-liners ("Add a task.", "Complete a task.").

The gap between the two is the concrete value of writing the KDoc as a prompt rather than as an
afterthought. That delta is the headline of the write-up.

## The dataset

`dataset.json` holds 68 intents across six goals: the five functions plus a set of deliberately
out-of-scope / ambiguous requests that should map to nothing. The phrasings were generated by
several independent authors and span registers on purpose — terse, verbose, polite, abrupt, slang,
indirect, and with typos — so the test isn't biased toward one way of asking. Each item carries its
expected function and, where relevant, the parameters that should be extracted.

## Running it

Needs a Gemini API key (free AI Studio tier is enough) in `gemini.api.key` at the repo root
(git-ignored). Standard-library Python only.

```bash
cd tools/reliability-harness
python3 harness.py --dataset dataset.json --out report.json \
    --model gemini-2.5-flash --variants rich,terse
```

Options: `--model` (any function-calling Gemini model), `--variants` (`rich`, `terse`, or both),
`--limit N` (first N intents, for a quick check), `--sleep` (pacing between calls). The run writes a
full per-item report to the `--out` JSON and prints a summary, including the rich-vs-terse delta. It
backs off automatically on rate limits.

## How the scoring works, and its limits

Function scoring is exact (predicted name vs expected, with "no call" as the expected answer for
out-of-scope intents). Parameter scoring is deliberately lenient where exactness would be unfair: a
title matches on a normalized substring, a dependency matches whether the model passed a title or an
id, a due date counts if one was extracted (date normalization varies), and priority/recurrence
match case-insensitively. The numbers therefore reflect *useful* routing, not string-identical
output. Calls run at temperature 0 for repeatability, but a hosted model is not perfectly
deterministic, so treat small differences between runs as noise.

This harness drives the Gemini API directly with the same function metadata the on-device agent
would see, which makes the selection/extraction measurement fast and reproducible. End-to-end
execution against the running app is verified separately (`adb shell cmd app_function
execute-app-function`, and the official Testing Agent).
